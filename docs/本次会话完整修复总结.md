## 修复的问题清单

### 问题1：科学计数法问题 ✅
**现象**：导入时卡号显示为 `6.21700123456789e+18`
**根本原因**：pandas 读取 Excel 时自动将长数字转换为 float64
**解决方案**：添加 `dtype={'卡号': str}` 参数
**修改文件**：`backend/services/case_card_service.py` 第288行

### 问题2：卡类型验证过于严格 ✅
**现象**：测试数据使用 value 格式（如"c15q"）被拒绝
**根本原因**：代码只支持 label 格式（如"入款卡"）
**解决方案**：同时支持 label 和 value 两种格式
**修改文件**：`backend/services/case_card_service.py` 第344-355行

### 问题3：数据库字段缺失（导入） ✅
**现象**：`Unknown column 'import_task_id' in 'field list'`
**根本原因**：案件数据库未迁移，缺少 import_task_id 字段
**解决方案**：动态检查字段是否存在，自动适配
**修改文件**：`backend/services/case_card_service.py` 第370-395行

### 问题4：卡类型字段设计错误 ✅
**现象**：card_type 存储的是 value（如"c15q"）
**根本原因**：应该存储 label（如"疑似员工卡"）
**解决方案**：修改映射关系，存储 label 而不是 value
**修改文件**：`backend/services/case_card_service.py` 第296-362行

### 问题5：数据库字段缺失（查询） ✅
**现象**：查询列表返回 500 错误
**根本原因**：SELECT 语句包含 import_task_id 字段，但数据库没有
**解决方案**：查询时也动态检查字段是否存在
**修改文件**：`backend/services/case_card_service.py` 第58-137行

### 问题6：代码语法错误 ✅
**现象**：get_case_card 方法中有重复的 WHERE 子句
**根本原因**：修改代码时产生的重复行
**解决方案**：删除重复的 WHERE 子句
**修改文件**：`backend/services/case_card_service.py` 第131行

### 问题7：卡类型必填验证 ✅
**现象**：卡类型可以为空，不符合业务需求
**根本原因**：Schema 中 card_type 是 Optional，导入时允许 NULL
**解决方案**：Schema 改为必填，导入时验证并拒绝空值
**修改文件**：
- `backend/schemas/case_card.py` 第10行
- `backend/services/case_card_service.py` 第341-368行

### 问题8：路由顺序冲突 ✅
**现象**：访问导入任务接口报错 "card_id 类型错误"
**根本原因**：具体路径 `import-tasks` 定义在参数化路径 `{card_id}` 之后
**解决方案**：调整路由顺序，具体路径在参数化路径之前
**修改文件**：`backend/api/v1/case_cards.py` 第189-330行

## 技术要点总结

### 1. pandas 类型推断问题
- pandas 会自动推断列类型
- 长数字会被转换为 float64，导致精度丢失
- 解决方法：使用 `dtype` 参数明确指定类型

### 2. 严格的导入策略
- 银行名称未匹配 → 设为 NULL，继续导入
- 卡类型为空 → ❌ 拒绝导入（必填）
- 卡类型无效 → ❌ 拒绝导入（必须是有效的 label 或 value）
- 只有卡号缺失或重复时才拒绝

### 3. 向后兼容性设计
- 动态检查数据库字段是否存在
- 支持多种输入格式（label 和 value）
- 不强制要求立即迁移

### 4. 数据库设计规范
- card_type 字段存储 d_label（显示名称）
- 不存储 d_value（内部代号）
- 便于查询和显示
