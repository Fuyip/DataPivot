# 路由顺序冲突修复说明

## 修复日期
2026-03-09

## 问题描述

用户反馈：**path.card_id: Input should be a valid integer, unable to parse string as an integer 获取导入任务失败。**

访问 `/api/v1/cases/{case_id}/case-cards/import-tasks` 时报错，提示 `card_id` 参数类型错误。

---

## 问题原因

FastAPI 路由匹配是按照定义顺序进行的。在 `case_cards.py` 中：

**原来的路由顺序**：
```python
# 第189行：通用列表路由
@router.get("/{case_id}/case-cards")

# 第212行：单条记录路由（包含路径参数 {card_id}）
@router.get("/{case_id}/case-cards/{card_id}")

# 第287行：导入任务列表路由
@router.get("/{case_id}/case-cards/import-tasks")
```

**问题分析**：
当访问 `/case-cards/import-tasks` 时，FastAPI 会按顺序匹配：
1. 先匹配 `/{case_id}/case-cards` - 不匹配（路径更长）
2. 再匹配 `/{case_id}/case-cards/{card_id}` - **匹配成功**！
3. 将 `"import-tasks"` 当作 `card_id` 参数
4. 尝试将字符串 `"import-tasks"` 转换为整数 → 类型错误

**根本原因**：更具体的路由（`import-tasks`）定义在更通用的路由（`{card_id}`）之后。

---

## 解决方案

将具体路径的路由移到参数化路由之前，确保 FastAPI 先匹配具体路径。

**修改后的路由顺序**：
```python
# 第189行：导入任务列表路由（具体路径）
@router.get("/{case_id}/case-cards/import-tasks")

# 第206行：删除导入任务路由（具体路径）
@router.delete("/{case_id}/case-cards/import-tasks/{task_id}")

# 第231行：通用列表路由
@router.get("/{case_id}/case-cards")

# 第254行：单条记录路由（参数化路径）
@router.get("/{case_id}/case-cards/{card_id}")
```

**文件**：`backend/api/v1/case_cards.py`

## FastAPI 路由匹配规则

### 规则说明
FastAPI 按照路由定义的顺序进行匹配，遵循"先定义先匹配"原则。

### 最佳实践
路由定义顺序应该从具体到通用：

```python
# ✅ 正确顺序
@router.get("/users/me")           # 最具体
@router.get("/users/active")       # 具体
@router.get("/users/{user_id}")    # 通用（参数化）
@router.get("/users")              # 最通用

# ❌ 错误顺序
@router.get("/users/{user_id}")    # 会匹配所有 /users/xxx
@router.get("/users/me")           # 永远不会被匹配到
```

### 路由优先级
1. **静态路径** > **参数化路径**
   - `/users/me` 优先于 `/users/{user_id}`
2. **更长的路径** > **更短的路径**
   - `/users/me/profile` 优先于 `/users/me`
3. **先定义** > **后定义**
   - 相同优先级时，先定义的先匹配
