# DataPivot 权限控制文档

## 1. 系统概述

DataPivot 采用三层权限控制架构，确保系统安全性和灵活性：

1. **全局角色层**：定义用户在系统中的基础权限级别
2. **案件权限层**：控制用户对特定案件的访问权限
3. **细粒度权限层**：预定义的功能权限系统（可扩展）

### 权限层级关系

```
全局角色 (User.role)
├── super_admin (超级管理员) - 拥有所有权限，无限制
├── admin (管理员) - 可管理用户和案件，受限于创建关系
└── user (普通用户) - 只能访问被授权的案件

案件权限 (UserCasePermission)
├── admin (案件管理员) - 完全控制案件
├── write (读写权限) - 可读写案件数据
└── read (只读权限) - 只能查看案件数据
```

## 2. 全局角色定义

### 2.1 超级管理员 (super_admin)

**权限范围**：
- ✓ 所有系统功能无限制访问
- ✓ 管理所有用户（创建、修改、删除、角色变更）
- ✓ 管理所有案件（创建、编辑、删除、归档）
- ✓ 分配任意案件的权限给任意用户
- ✓ 初始化系统权限配置
- ✓ 查看系统日志和配置

**使用场景**：系统管理员、技术负责人

### 2.2 管理员 (admin)

**权限范围**：
- ✓ 创建普通用户（仅限 user 角色）
- ✓ 管理自己创建的用户（修改、删除）
- ✓ 创建案件（自动成为案件管理员）
- ✓ 管理自己是管理员的案件（编辑、归档）
- ✓ 软删除自己创建的案件（标记 is_deleted = true）
- ✓ 为自己创建的用户分配案件权限
- ✗ 不能修改用户角色
- ✗ 不能彻底删除案件（物理删除）
- ✗ 不能管理其他管理员创建的用户
- ✗ 不能访问自己无权限的案件

**使用场景**：部门负责人、项目经理

### 2.3 普通用户 (user)

**权限范围**：
- ✓ 访问被授权的案件（根据案件权限级别）
- ✓ 修改自己的个人信息
- ✗ 不能创建用户
- ✗ 不能创建案件
- ✗ 不能分配权限（除非是案件管理员）

**使用场景**：数据分析师、普通操作员

## 3. 案件权限管理

### 3.1 权限级别定义

| 权限级别 | 代码值 | 权限等级 | 说明 |
|---------|--------|---------|------|
| admin | admin | Level 3 | 案件管理员，拥有完全控制权 |
| write | write | Level 2 | 读写权限，可修改案件数据 |
| read | read | Level 1 | 只读权限，仅可查看案件数据 |

**权限检查规则**：用户实际权限级别 >= 所需权限级别

### 3.2 案件管理员 (admin)

**权限范围**：
- ✓ 查看案件所有信息
- ✓ 编辑案件基本信息
- ✓ 归档/恢复案件
- ✓ 导入/导出案件数据
- ✓ 分析案件数据
- ✓ 分配/修改/撤销其他用户的案件权限
- ✓ 查看案件权限列表

**获取方式**：
- 创建案件时自动成为案件管理员
- 由 super_admin 或其他案件管理员授予

### 3.3 读写权限 (write)

**权限范围**：
- ✓ 查看案件所有信息
- ✓ 导入/导出案件数据
- ✓ 修改案件数据
- ✓ 分析案件数据
- ✗ 不能编辑案件基本信息
- ✗ 不能归档案件
- ✗ 不能管理权限

**获取方式**：由案件管理员或 super_admin 授予

### 3.4 只读权限 (read)

**权限范围**：
- ✓ 查看案件基本信息
- ✓ 查看案件数据
- ✓ 导出案件数据（仅查看）
- ✗ 不能修改任何数据
- ✗ 不能导入数据
- ✗ 不能管理权限

**获取方式**：由案件管理员或 super_admin 授予

**权限检查逻辑**：
- super_admin：无限制
- admin：只能操作自己创建的用户（created_by = current_user.id）
- user：只能查看和修改自己的信息

### 5.2 案件管理 API

**基础路径**：`/api/v1/cases`

| 端点 | 方法 | 权限要求 | 说明 |

| `/cases/{id}/archive` | POST | 案件 admin 权限 | 归档案件 |
| `/cases/{id}/restore` | POST | 案件 admin 权限 | 恢复案件 |

**权限检查逻辑**：
- super_admin：可访问所有案件，可彻底删除（物理删除）
- admin：可访问自己是管理员的案件，删除时仅软删除（is_deleted = true）
- user：只能访问被授权的案件（通过 user_case_permissions 表）

## 6. 权限检查流程

### 6.1 核心权限检查函数

**文件路径**：`backend/services/case_service.py`

```python
def check_case_permission(db: Session, user_id: int, case_id: int, required_level: str = "read") -> bool:
    """
    检查用户是否有指定案件的权限

    参数:
        db: 数据库会话
        user_id: 用户ID
        case_id: 案件ID
        required_level: 需要的权限级别 (read/write/admin)

    返回:
        bool: 用户权限级别 >= 需要的权限级别
    """
    permission_levels = {"read": 1, "write": 2, "admin": 3}
    required_level_value = permission_levels.get(required_level, 1)

    # 查询用户对该案件的权限
    permission = db.query(UserCasePermission).filter(
        UserCasePermission.user_id == user_id,
        UserCasePermission.case_id == case_id
    ).first()

    if not permission:
        return False

    user_level_value = permission_levels.get(permission.role, 0)
    return user_level_value >= required_level_value
```

### 6.2 依赖注入权限检查

**文件路径**：`backend/services/case_service.py`

```python
def require_case_permission(required_level: str = "read"):
    """
    案件权限检查依赖注入装饰器

    使用示例:
        @router.get("/cases/{case_id}")
        def get_case(
            case_id: int,
            current_user = Depends(require_case_permission("read"))
        ):
            # 业务逻辑
    """
    def check_permission(
        case_id: int,
        current_user = Depends(get_current_active_user),
        db: Session = Depends(get_system_db)
    ):
        # super_admin 和 admin 自动拥有所有权限
        if current_user.role in ["super_admin", "admin"]:
            return current_user

        # 普通用户需要检查具体权限
        has_permission = check_case_permission(db, current_user.id, case_id, required_level)
        if not has_permission:
            raise HTTPException(status_code=403, detail="权限不足")

        return current_user

    return check_permission
```

### 6.3 权限检查流程图

```
用户请求 API
    ↓
验证 JWT Token → 获取 current_user
    ↓
检查用户是否激活 (is_active)
    ↓
判断全局角色:
    ├─ super_admin → ✓ 允许访问
    ├─ admin → 检查是否满足 admin 特定规则
    └─ user → 检查案件权限
         ↓
    查询 user_case_permissions 表
         ↓
    比较权限级别: user_level >= required_level
         ↓
    ✓ 允许访问 / ✗ 返回 403 错误
```

## 7. 数据库表结构

### 7.1 用户表 (users)

**文件路径**：`sql/init/001_create_users_table.sql`

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    hashed_password VARCHAR(255) NOT NULL,
    full_name VARCHAR(100),
    email VARCHAR(100) UNIQUE,
    role VARCHAR(20) DEFAULT 'user',           -- super_admin/admin/user
    is_active TINYINT(1) DEFAULT 1,
    created_by INT,                             -- 创建人ID
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES users(id)
);
```

**关键字段说明**：
- `role`：全局角色，决定用户的基础权限
- `created_by`：记录创建人，用于 admin 权限控制
- `is_active`：用户状态，禁用后无法登录

### 7.2 案件表 (cases)

**文件路径**：`sql/init/002_create_cases_tables.sql`

```sql
CREATE TABLE cases (
    id INT PRIMARY KEY AUTO_INCREMENT,
    case_name VARCHAR(200) UNIQUE NOT NULL,
    case_code VARCHAR(100) UNIQUE NOT NULL,
    database_name VARCHAR(100) UNIQUE NOT NULL,
    description TEXT,
    status VARCHAR(20) DEFAULT 'active',       -- active/archived/closed
    is_active BOOLEAN DEFAULT TRUE,
    is_deleted BOOLEAN DEFAULT FALSE,
    created_by INT,                             -- 创建人ID
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES users(id)
);
```

**关键字段说明**：
- `created_by`：案件创建人，创建时自动获得 admin 权限
- `status`：案件状态，影响访问控制
- `is_deleted`：软删除标记（admin 删除案件时设置为 true，super_admin 可彻底删除）

### 7.3 用户案件权限表 (user_case_permissions)

**文件路径**：`sql/init/002_create_cases_tables.sql`

```sql
CREATE TABLE user_case_permissions (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    case_id INT NOT NULL,
    role VARCHAR(20) DEFAULT 'user',           -- admin/user (映射到 admin/write/read)
    granted_by INT,                             -- 授权人ID
    granted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY unique_user_case (user_id, case_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (case_id) REFERENCES cases(id) ON DELETE CASCADE,
    FOREIGN KEY (granted_by) REFERENCES users(id)
);
```

**关键字段说明**：
- `role`：案件内角色（admin/user），通过 permission_levels 映射到 read/write/admin
- `granted_by`：记录授权人，用于审计
- `unique_user_case`：确保一个用户对一个案件只有一条权限记录

**注意事项**：
- 当前实现中，`role` 字段存储 "admin" 或 "user"
- 权限级别通过代码映射：read(1) < write(2) < admin(3)
- 建议未来扩展：将 role 字段改为 permission_level，直接存储 read/write/admin

### 7.4 权限定义表 (permissions)

**文件路径**：`sql/init/002_create_cases_tables.sql`

```sql
CREATE TABLE permissions (
    id INT PRIMARY KEY AUTO_INCREMENT,
    code VARCHAR(50) UNIQUE NOT NULL,          -- 权限代码，如 user.create
    name VARCHAR(100) NOT NULL,                -- 权限名称
    description TEXT,                          -- 权限描述
    category VARCHAR(50),                      -- 权限分类：user/case/data/system
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**预定义权限分类**：

| 分类 | 权限代码示例 | 说明 |
|------|------------|------|
| user | user.create, user.read, user.update, user.delete, user.manage_role | 用户管理权限 |
| case | case.create, case.read, case.update, case.delete, case.archive, case.manage_permission | 案件管理权限 |
| data | data.import, data.export, data.read, data.update, data.delete, data.analyze | 数据操作权限 |
| system | system.config, system.log, system.backup | 系统管理权限 |

**初始化接口**：`POST /api/v1/permissions/init`（仅 super_admin 可调用）

router.beforeEach((to, from, next) => {

})

## 9. 安全建议

### 9.1 密码安全

- ✓ 使用 BCrypt 加密存储密码（当前已实现）
- ✓ 强制密码复杂度要求（建议实施）
- ✓ 定期强制修改密码（建议实施）
- ⚠️ **重要**：生产环境必须立即修改默认管理员密码（admin/admin123）

### 9.2 Token 安全

- ✓ 使用 JWT Token 进行身份验证（当前已实现）
- ✓ Token 过期时间：1440 分钟（可配置）
- ✓ 建议实施 Token 刷新机制
- ✓ 建议实施 Token 黑名单（用户登出后失效）

### 9.3 权限审计

- ✓ 记录权限授予人（granted_by 字段）
- ✓ 记录权限授予时间（granted_at 字段）
- ✓ 建议实施操作日志系统
- ✓ 建议定期审查权限分配情况

### 9.4 最小权限原则

- 默认授予最小必要权限
- 普通用户默认只有 read 权限
- 根据实际需要逐步提升权限
- 定期清理不再需要的权限

### 9.5 防护措施

- ✓ API 层面强制权限检查（当前已实现）
- ✓ 前端隐藏无权限功能（当前已实现）
- ✓ 数据库外键级联删除（防止孤立数据）
- ✓ 软删除机制（is_deleted 字段，admin 删除案件时使用）
- ✓ 建议实施 API 访问频率限制
- ✓ 建议实施异常登录检测

**软删除与硬删除说明**：
- **软删除**：设置 `is_deleted = true`，数据保留在数据库中，可恢复
  - admin 删除案件时使用软删除
  - 适用于需要数据审计和恢复的场景
- **硬删除**：从数据库中物理删除记录，不可恢复
  - 仅 super_admin 可执行硬删除
  - 用于彻底清理不需要的数据

## 11. 常见问题

### Q1: admin 和 super_admin 的主要区别是什么？

**A**:
- `super_admin`：系统最高权限，无任何限制
- `admin`：受限的管理员，只能管理自己创建的用户和自己是管理员的案件

### Q2: 案件权限中的 write 和 read 如何映射？

**A**: 当前实现中，数据库存储的是 `role` 字段（admin/user），通过代码映射：
- `admin` → Level 3（完全控制）
- `write` → Level 2（读写，当前映射到 user 角色）
- `read` → Level 1（只读，当前映射到 user 角色）

建议未来优化：直接在数据库中存储 permission_level 字段。

### Q3: 如何给用户分配案件权限？

**A**:
1. 确保操作者是案件管理员或 super_admin
2. 如果是 admin 角色，目标用户必须由自己创建
3. 调用 `POST /api/v1/cases/{case_id}/permissions` 接口
4. 指定 user_id 和 role（admin/user）

### Q4: 用户被删除后，其权限记录会怎样？

**A**: 由于设置了 `ON DELETE CASCADE`，用户被删除时：
- 其在 `user_case_permissions` 表中的记录会自动删除
- 其创建的案件不会被删除（created_by 字段会保留）

### Q5: admin 和 super_admin 在删除案件时有什么区别？

**A**:
- `admin`：只能软删除自己创建的案件（设置 `is_deleted = true`），数据保留可恢复
- `super_admin`：可以彻底删除任何案件（物理删除），数据不可恢复

### Q6: 如何实现更细粒度的权限控制？

**A**: 可以使用 `permissions` 表中的预定义权限系统：
1. 调用 `POST /api/v1/permissions/init` 初始化权限
2. 创建 `role_permissions` 关联表
3. 在 API 中检查用户是否拥有特定权限代码

### 示例 1：检查用户是否可以创建案件

```python
def can_create_case(user: User) -> bool:
    """检查用户是否可以创建案件"""
    return user.role in ["super_admin", "admin"]
```

### 示例 2：检查用户是否可以删除另一个用户

```python
def can_delete_user(operator: User, target_user: User) -> bool:
    """检查操作者是否可以删除目标用户"""
    if operator.role == "super_admin":
        return True
    if operator.role == "admin":
        return target_user.created_by == operator.id
    return False
```

### 示例 3：获取用户可访问的案件列表

```python
def get_accessible_cases(db: Session, user: User) -> List[Case]:
    """获取用户可访问的案件列表"""
    if user.role == "super_admin":
        # super_admin 可以访问所有案件
        return db.query(Case).filter(Case.is_deleted == False).all()
    elif user.role == "admin":
        # admin 可以访问自己是管理员的案件
        case_ids = db.query(UserCasePermission.case_id).filter(
            UserCasePermission.user_id == user.id,
            UserCasePermission.role == "admin"
        ).all()
        return db.query(Case).filter(
            Case.id.in_([c[0] for c in case_ids]),
            Case.is_deleted == False
        ).all()
    else:
        # user 只能访问被授权的案件
        case_ids = db.query(UserCasePermission.case_id).filter(
            UserCasePermission.user_id == user.id
        ).all()
        return db.query(Case).filter(
            Case.id.in_([c[0] for c in case_ids]),
            Case.is_deleted == False
        ).all()
```
