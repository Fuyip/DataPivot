# 权限显示"未知"问题修复总结

## 问题描述

超级管理员分配案件权限后，权限列表和案件列表中显示"未知"，且无法修改权限。

## 根本原因

### 1. 字段名不匹配
- **前端发送**: `permission_level` 字段
- **后端期望**: `role` 字段（Schema定义）
- **后端返回**: `role` 字段
- **前端期望**: `permission_level` 字段

### 2. 数据库设计与前端UI不一致
- **数据库**: `role` 字段只有两个值 (`admin`, `user`)
- **前端UI**: 提供三个选项（管理员、读写、只读）
- **映射关系**:
  - `admin` → 管理员
  - `user` → 读写/只读（无法区分）

## 修复方案

### 已完成的修复

#### 1. 后端API修改 (backend/api/v1/case_permissions.py)

**修改内容**:
- 将接口参数从 Schema 改为 `dict`，兼容前后端字段名
- 同时支持 `role` 和 `permission_level` 字段
- 返回数据同时包含两个字段，确保兼容性
- 将前端的 `read`/`write` 映射为数据库的 `user`
- 统一处理 `super_admin` 权限检查逻辑

**修改的接口**:
1. `GET /cases/{case_id}/permissions` - 获取权限列表
2. `POST /cases/{case_id}/permissions` - 创建权限
3. `PUT /cases/{case_id}/permissions/{permission_id}` - 更新权限
4. `DELETE /cases/{case_id}/permissions/{permission_id}` - 删除权限
5. `GET /cases/users/{user_id}/cases` - 获取用户案件列表

**关键代码**:
```python
# 返回数据同时包含两个字段
{
    "role": perm.role,
    "permission_level": perm.role,  # 兼容前端
    ...
}

# 接收参数兼容两种字段名
role = permission_data.get("role") or permission_data.get("permission_level")

# 将前端三级权限映射为数据库两级权限
if role in ["read", "write"]:
    role = "user"
```

### 当前限制

由于数据库 `role` 字段只有 `admin` 和 `user` 两个值：
- ✅ 可以区分"管理员"权限
- ❌ 无法区分"读写"和"只读"权限
- 前端选择"读写"或"只读"时，后端都存储为 `user`
- 所有非管理员用户的权限在前端都会显示为"读写"

## 未来优化建议

### 方案1: 扩展数据库字段（推荐）

如需真正实现三级权限（admin/write/read），建议：

1. **修改数据库表结构**
   ```sql
   ALTER TABLE user_case_permissions
   MODIFY COLUMN role VARCHAR(20) DEFAULT 'read'
   COMMENT '案件内角色: admin/write/read';
   ```

2. **更新现有数据**
   ```sql
   -- 将现有的 user 权限统一改为 write
   UPDATE user_case_permissions
   SET role = 'write'
   WHERE role = 'user';
   ```

3. **更新权限检查逻辑**
   ```python
   # backend/services/case_service.py
   permission_levels = {"read": 1, "write": 2, "admin": 3}
   ```

4. **更新Schema验证**
   ```python
   # backend/schemas/case.py
   role: str = Field(default="read", pattern="^(admin|write|read)$")
   ```

### 方案2: 保持现状（简单）

如果不需要区分读写和只读权限：
- 前端UI只提供两个选项：管理员、普通用户
- 移除"读写"和"只读"的区分
- 简化权限管理逻辑
