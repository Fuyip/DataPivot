# 银行信息管理功能设计文档

## 1. 功能概述

银行信息管理功能用于维护 DataPivot 系统中的银行卡 BIN 码库（`bank_bin` 表）和银行名称映射表（`sy_bank` 表）。该功能实现了基于角色的权限控制和审批流程，确保数据修改的安全性和可追溯性。

## 2. 功能目标

- 提供 `bank_bin` 表和 `sy_bank` 表的可视化管理界面
- 实现基于角色的权限控制（管理员和超级管理员）
- 建立变更申请和审批流程
- 支持批量操作和数据导入导出
- 记录所有变更历史，确保可追溯性

## 3. 权限设计

### 3.1 角色定义

| 角色 | 权限 |
|------|------|
| 普通用户 (user) | 无权访问此功能 |
| 管理员 (admin) | 可以查看数据、提交变更申请（新增、修改、删除） |
| 超级管理员 (super_admin) | 拥有管理员所有权限 + 审批变更申请 + 直接修改数据 |

### 3.2 权限矩阵

| 操作 | 管理员 | 超级管理员 |
|------|--------|-----------|
| 查看 bank_bin 数据 | ✅ | ✅ |
| 查看 sy_bank 数据 | ✅ | ✅ |
| 提交新增申请 | ✅ | ✅ |
| 提交修改申请 | ✅ | ✅ |
| 提交删除申请 | ✅ | ✅ |
| 审批变更申请 | ❌ | ✅ |
| 拒绝变更申请 | ❌ | ✅ |
| 直接修改数据（无需审批） | ❌ | ✅ |
| 查看变更历史 | ✅ | ✅ |
| 导入数据 | ✅ | ✅ |
| 导出数据 | ✅ | ✅ |

## 4. 数据表设计

### 4.1 bank_bin 表（现有表）

银行卡 BIN 码库，用于根据卡号前缀匹配银行信息。

| 字段 | 类型 | 说明 |
|------|------|------|
| bin | VARCHAR | BIN 码（卡号前缀） |
| bin_len | INT | BIN 码长度 |
| card_len | INT | 完整卡号长度 |
| bank_name | VARCHAR | 银行名称 |

### 4.2 sy_bank 表（现有表）

银行名称映射表，用于统一不同系统的银行名称。

| 字段 | 类型 | 说明 |
|------|------|------|
| from_bank | VARCHAR | 原始银行名称 |
| to_bank | VARCHAR | 标准银行名称 |
| sys | VARCHAR | 系统标识（如 'jz'） |

### 4.3 bank_info_change_requests 表（新增表）

银行信息变更申请表，记录所有变更申请和审批流程。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键 |
| table_type | ENUM | 表类型：bank_bin / sy_bank |
| change_type | ENUM | 变更类型：create / update / delete |
| old_data | TEXT | 原始数据（JSON 格式） |
| new_data | TEXT | 新数据（JSON 格式） |
| reason | VARCHAR(500) | 变更原因 |
| status | ENUM | 状态：pending / approved / rejected / executed |
| created_by | INT | 申请人 ID（外键关联 users 表） |
| created_at | DATETIME | 申请时间 |
| reviewed_by | INT | 审批人 ID（外键关联 users 表） |
| reviewed_at | DATETIME | 审批时间 |
| review_comment | VARCHAR(500) | 审批意见 |
| executed_at | DATETIME | 执行时间 |

## 5. 业务流程

### 5.1 管理员提交变更申请流程

```
1. 管理员在界面上进行数据操作（新增/修改/删除）
2. 填写变更原因
3. 提交变更申请
4. 系统创建变更申请记录（状态：pending）
5. 等待超级管理员审批
```

### 5.2 超级管理员审批流程

```
1. 超级管理员查看待审批的变更申请列表
2. 查看变更详情（原始数据 vs 新数据）
3. 选择批准或拒绝
   - 批准：
     a. 更新申请状态为 approved
     b. 记录审批人和审批时间
     c. 执行数据库变更操作（作为事务）
     d. 更新申请状态为 executed
     e. 记录执行时间
   - 拒绝：
     a. 更新申请状态为 rejected
     b. 记录审批人、审批时间和拒绝原因
```

### 5.3 超级管理员直接修改流程

```
1. 超级管理员在界面上进行数据操作
2. 选择"直接修改"（跳过审批）
3. 系统直接执行数据库变更
4. 创建变更记录（状态：executed，无需审批）
```

## 6. 功能模块

### 6.1 bank_bin 管理

#### 6.1.1 列表展示
- 分页展示所有 BIN 码记录
- 支持按 BIN 码、银行名称搜索
- 支持按 BIN 长度、卡长度筛选
- 显示字段：BIN 码、BIN 长度、卡长度、银行名称

#### 6.1.2 新增 BIN 码
- 表单字段：
  - BIN 码（必填，数字）
  - BIN 长度（必填，自动计算）
  - 卡长度（必填，下拉选择：16/17/18/19）
  - 银行名称（必填）
- 管理员：提交申请
- 超级管理员：可选择直接添加或提交申请

#### 6.1.3 修改 BIN 码
- 可修改字段：BIN 码、BIN 长度、卡长度、银行名称
- 管理员：提交修改申请
- 超级管理员：可选择直接修改或提交申请

#### 6.1.4 删除 BIN 码
- 单条删除或批量删除
- 管理员：提交删除申请
- 超级管理员：可选择直接删除或提交申请

#### 6.1.5 导入导出
- 导入：支持 Excel 批量导入（管理员提交申请，超级管理员可直接导入）
- 导出：导出当前筛选结果为 Excel
- 模板下载：提供标准导入模板

### 6.2 sy_bank 管理

#### 6.2.1 列表展示
- 分页展示所有银行名称映射记录

#### 6.2.3 修改映射
- 可修改字段：原始银行名称、标准银行名称、系统标识
- 管理员：提交修改申请
- 超级管理员：可选择直接修改或提交申请

#### 6.2.4 删除映射
- 单条删除或批量删除
- 管理员：提交删除申请
- 超级管理员：可选择直接删除或提交申请

- 模板下载：提供标准导入模板

### 6.3 变更申请管理

#### 6.3.1 待审批列表（超级管理员）
- 显示所有待审批的变更申请

## 9. 技术实现要点

### 9.1 事务处理

批准变更申请时，需要确保以下操作在同一事务中完成：
1. 更新申请状态为 approved
2. 执行数据库变更（INSERT/UPDATE/DELETE）
3. 更新申请状态为 executed
4. 记录执行时间

如果任何步骤失败，整个事务回滚。

### 9.2 数据验证

- BIN 码必须是纯数字
- BIN 长度必须与 BIN 码实际长度一致
- 卡长度必须在合理范围内（16-19）
- 银行名称不能为空
- 导入数据时进行格式验证和重复检查

### 9.3 安全性

- 所有接口都需要登录认证
- 基于角色的权限控制（RBAC）
- 记录所有操作日志
- 敏感操作需要二次确认

### 9.4 性能优化

- 列表查询使用分页
- 批量操作使用事务批处理
- 导入大量数据时使用异步任务
