# 导入功能兼容性修复说明

## 修复日期
2026-03-09

## 问题描述

在测试导入功能时发现两个问题：

### 问题1：卡类型验证过于严格
```
第2行 - 卡号: 6217001234567890123 - 无效的卡类型: c15q
```

**原因**：代码期望卡类型是 label（如"入款卡"），但测试数据中使用的是 value（如"c15q"）

### 问题2：数据库字段缺失
```
Unknown column 'import_task_id' in 'field list'
```

**原因**：案件数据库中的 case_card 表还没有添加 import_task_id 字段

---

## 解决方案

### 修复1：卡类型验证兼容性

**文件**：`backend/services/case_card_service.py`

**修改前**：
```python
# 只支持 label，不支持 value
card_type_label = str(row.get('卡类型', '')).strip()
card_type_value = card_type_map.get(card_type_label)
if not card_type_value:
    # 拒绝导入
    error_count += 1
    continue
```

**修改后**：
```python
# 同时支持 label 和 value
card_type_input = str(row.get('卡类型', '')).strip()
card_type_value = None
if card_type_input:
    # 先尝试作为 label 查找
    card_type_value = card_type_map.get(card_type_input)
    # 如果找不到，检查是否本身就是 value
    if not card_type_value:
        valid_values = [ct['value'] for ct in card_types]
        if card_type_input in valid_values:
            card_type_value = card_type_input
        else:
            # 既不是 label 也不是 value，设为 None（不拒绝导入）
            card_type_value = None
```

**改进点**：
1. 支持 label 格式（如"入款卡"）
2. 支持 value 格式（如"c15q"）
3. 无效的卡类型不再拒绝导入，而是设为 NULL

### 修复2：数据库字段兼容性

**文件**：`backend/services/case_card_service.py`

**修改前**：
```python
# 直接插入 import_task_id，如果字段不存在会报错
insert_sql = """
    INSERT INTO case_card (..., import_task_id)
    VALUES (..., :import_task_id)
"""
```

**修改后**：
```python
# 检查表是否有 import_task_id 字段
check_column_sql = """
    SELECT COUNT(*)
    FROM information_schema.COLUMNS
    WHERE TABLE_SCHEMA = DATABASE()
    AND TABLE_NAME = 'case_card'
    AND COLUMN_NAME = 'import_task_id'
"""
has_import_task_id = db.execute(text(check_column_sql)).scalar() > 0

# 根据字段是否存在，使用不同的 SQL
if has_import_task_id:
    card_data["import_task_id"] = task_id
    insert_sql = """
        INSERT INTO case_card (..., import_task_id)
        VALUES (..., :import_task_id)
    """
else:
    insert_sql = """
        INSERT INTO case_card (...)
        VALUES (...)
    """
```

**改进点**：
1. 动态检查字段是否存在
2. 兼容已迁移和未迁移的数据库
3. 不会因为字段缺失而导致导入失败

## 设计理念

### 宽容的导入策略

导入功能采用"宽容"策略，尽量不拒绝数据：

| 问题类型 | 处理方式 | 理由 |
|---------|---------|------|
| 卡号已存在 | 跳过，记录错误 | 避免重复数据 |
| 银行名称未匹配 | 设为 NULL，继续导入 | 可以后续手动补充 |
| 卡类型无效 | 设为 NULL，继续导入 | 可以后续手动补充 |
| 卡号为空 | 跳过，记录错误 | 卡号是必需字段 |
| 数据库字段缺失 | 自动适配 | 兼容不同版本 |

**原则**：只有关键字段（如卡号）缺失或重复时才拒绝，其他问题都设为 NULL 并继续导入。

### 向后兼容性

系统设计考虑了向后兼容：

1. **数据库迁移**：新功能不强制要求所有案件数据库立即迁移
2. **字段检查**：动态检查字段是否存在，自动适配
3. **数据格式**：支持多种输入格式（label/value）
4. **错误处理**：详细记录错误，但不阻塞整个导入流程

## 数据库迁移建议

虽然系统已经兼容未迁移的数据库，但建议在合适的时候为所有案件数据库添加 import_task_id 字段：

```sql
-- 为案件数据库添加 import_task_id 字段
ALTER TABLE case_card
ADD COLUMN import_task_id INT DEFAULT NULL COMMENT '导入任务ID',
ADD INDEX idx_import_task_id (import_task_id);
```

**迁移优先级**：
- 高优先级：经常使用的案件数据库
- 中优先级：偶尔使用的案件数据库
- 低优先级：历史归档的案件数据库

**迁移时机**：
- 可以逐步迁移，不需要一次性完成
- 系统会自动适配已迁移和未迁移的数据库
- 不影响现有功能的使用
