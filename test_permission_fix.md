# 权限显示问题修复说明

## 问题分析

### 根本原因
前后端字段名不匹配导致权限显示为"未知"：

1. **前端发送字段**：`permission_level` (admin/write/read)
2. **后端期望字段**：`role` (admin/user)
3. **后端返回字段**：`role`
4. **前端期望接收**：`permission_level`

### 数据库设计
- `user_case_permissions` 表的 `role` 字段只存储两个值：
  - `admin` - 案件管理员
  - `user` - 普通用户（对应前端的 write/read）

## 修复方案

### 后端修改 (backend/api/v1/case_permissions.py)

#### 1. 获取权限列表接口
- 返回数据同时包含 `role` 和 `permission_level` 字段
- 两个字段值相同，兼容前后端

#### 2. 创建权限接口
- 接收 `dict` 类型参数，兼容 `role` 和 `permission_level` 字段
- 将前端的 `read`/`write` 映射为数据库的 `user`
- 返回数据同时包含两个字段

#### 3. 更新权限接口
- 接收 `dict` 类型参数，兼容 `role` 和 `permission_level` 字段
- 将前端的 `read`/`write` 映射为数据库的 `user`
- 返回数据同时包含两个字段

#### 4. 权限检查逻辑优化
- 所有接口统一处理 `super_admin` 权限
- `super_admin` 可以执行所有操作，无需额外检查

## 注意事项

### 当前限制
由于数据库 `role` 字段只有 `admin` 和 `user` 两个值：
- 前端选择"读写"或"只读"时，后端都会存储为 `user`
- 这意味着目前无法区分"读写"和"只读"权限
- 所有非管理员权限在数据库中都是 `user`

### 未来优化建议
如需真正实现三级权限（admin/write/read），需要：
1. 修改数据库表结构，将 `role` 字段改为支持三个值
2. 或者添加新的 `permission_level` 字段
3. 更新所有相关的权限检查逻辑
