# 三级权限实现测试报告

## 实施方案

在不迁移数据库的情况下，使用 SQLAlchemy 字段映射 + 应用层值转换实现三级权限。

## 实施内容

### 1. SQLAlchemy 字段映射

**文件**: `backend/models/case.py`

```python
# Python 属性名为 permission_level，但映射到数据库字段 role
permission_level = Column("role", String(20), default="user", comment="权限级别")
```

**效果**:
- Python 代码访问 `permission_level` 属性
- 数据库实际使用 `role` 字段
- SQLAlchemy 自动处理映射

### 2. 权限值转换函数

**文件**: `backend/services/case_service.py`

添加了两个转换函数：

```python
def map_permission_to_db(permission_level: str) -> str:
    """API值 -> 数据库值"""
    mapping = {
        "read": "user",    # 只读映射为 user
        "write": "user",   # 读写映射为 user
        "admin": "admin"   # 管理员保持 admin
    }
    return mapping.get(permission_level, "user")

def map_permission_from_db(role: str) -> str:
    """数据库值 -> API值"""
    if role == "admin":
        return "admin"
    return "write"  # user 统一返回 write
```

### 3. API 层修改

**文件**: `backend/api/v1/case_permissions.py`

- 创建权限时：使用 `map_permission_to_db()` 转换
- 读取权限时：使用 `map_permission_from_db()` 转换
- 返回数据时：保持原始请求的权限级别

**文件**: `backend/api/v1/cases.py`

- 创建案件时分配权限：使用 `map_permission_to_db("admin")`

## 方案限制

### 核心限制

由于数据库只有 `admin` 和 `user` 两个值，无法真正区分 `read` 和 `write`：

1. **写入时**:
   - API 接受: `read`, `write`, `admin`
   - 数据库存储: `user`, `user`, `admin`
   - read 和 write 都存储为 user

2. **读取时**:
   - 数据库值: `user`, `admin`
   - API 返回: `write`, `admin`
   - 所有 user 统一显示为 write

3. **实际效果**:
   - ✅ admin 权限: 完全控制
   - ✅ write 权限: 读写数据
   - ❌ read 权限: **实际上也是读写**（因为存储为 user）

### 用户体验影响

- 用户可以在前端选择"只读"权限
- 但实际上会被存储为"读写"权限
- 再次查看时会显示为"读写"
- **无法实现真正的只读权限控制**

## 总结

### 成功实现

✅ 字段映射: Python 使用 `permission_level`，数据库使用 `role`
✅ 值转换: API 支持三级权限值（read/write/admin）
✅ 向后兼容: 不需要修改数据库结构
✅ 代码统一: 所有代码使用 `permission_level` 属性

### 已知限制

⚠️ 无法真正区分 read 和 write 权限
⚠️ 只读权限实际上是读写权限
⚠️ 用户选择的权限可能与实际权限不一致

### 建议

如果需要真正的三级权限控制，必须：

**方案A**: 执行数据库迁移
- 将 `role` 字段改为 `permission_level`
- 支持三个值: `read`, `write`, `admin`
- 可以真正区分只读和读写

**方案B**: 添加额外的映射表
- 创建 `user_case_permission_mapping` 表
- 存储真实的三级权限
- 保持现有表结构不变

**方案C**: 接受当前限制
- 前端UI改为两级权限（读写、管理员）
- 移除"只读"选项
- 与数据库能力保持一致
